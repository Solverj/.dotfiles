## Chunked File Operations

When using `write` or `bash` for file-changing work, do not process large files in a single pass.
- Treat files larger than 200 lines or 8 KB as large for chunking purposes
- Split file updates into small chunks (target ~100-200 lines per operation)
- For `write`, prefer incremental writes or use `edit`/patch-based updates for each chunk
- For `bash` file-modifying commands, run per-file or per-section commands instead of repo-wide one-shot rewrites
- Allow full-file single-pass updates only when intermediate states would be invalid or externally consumed (generated files, lockfiles, formatter/codegen output)
- Verify each chunk before moving to the next chunk using a quick syntax/readback check and the smallest relevant test or lint scope
- If a chunk fails verification, stop immediately, fix that chunk, and re-run its verification before continuing
- After all chunks are complete, run one final end-to-end validation for the full changed scope

## JSON Tool Call Formatting Rules

**CRITICAL: When using write/edit tools, ensure all JSON is properly formatted to avoid parsing errors:**

1. **ALWAYS close all braces `}`** before ending a tool call
2. **ALWAYS close all brackets `]`** before ending a tool call  
3. **ALWAYS use double quotes `"`** for all string keys and values
4. **ALWAYS escape special characters** in strings:
   - Double quotes: `\"`
   - Backslashes: `\\`
   - Newlines: `\n` (use literal `\n`, not actual line breaks)
5. **NEVER truncate tool call arguments** - provide complete JSON
6. **For large content**, prefer multiple smaller tool calls over one large call

**Correct write tool example:**
```
write({
  filePath: "/path/to/file.ts",
  content: "export const greeting = \"Hello\";\n\nconsole.log(greeting);"
})
```

**Correct edit tool example:**
```
edit({
  filePath: "/path/to/file.ts",
  oldString: "console.log(\"old\");",
  newString: "console.log(\"new\");"
})
```

**Common mistakes to avoid:**
- ❌ Missing closing brace: `write({ filePath: "test.txt", content: "hello"`
- ❌ Unescaped quotes: `content: "He said "hello""`
- ❌ Actual newlines in strings: Use `\n` instead of pressing Enter
- ❌ Unescaped backslashes: Use `\\` in file paths like `C:\\Users\\...`

**When in doubt:**
- Use the `edit` tool instead of `write` for smaller changes (less content = less chance of errors)
- Break large files into multiple edits
- Always verify JSON is complete before sending
- If content has many quotes or special characters, be extra careful with escaping

## Output Format

**STRICT JSON OUTPUT REQUIRED**

All responses MUST be valid JSON matching the schema defined in `opencode.json`. Never output markdown, plain text, or any other format.

The `guided_json` parameter enforces this at the model level. If JSON parsing fails, the request will be retried automatically.

## JSON Output Schema

The build agent outputs strict JSON with these fields:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `status` | string | Yes | Overall status: success/partial/failed/in_progress |
| `step` | object | Yes | Current step being executed |
| `step.number` | integer | Yes | Step number from plan |
| `step.name` | string | Yes | Step name/description |
| `step.status` | string | Yes | Step status: pending/in_progress/completed/skipped/failed |
| `files_changed` | array | No | List of files modified in this step |
| `tests` | object | No | Test results if tests were run |
| `output` | string | Yes | Human-readable summary |
| `next_action` | string | Yes | What to do next: continue/ask_question/wait_for_approval/complete |
| `questions` | array | No | Questions for the user (if next_action is ask_question) |

## Rigor

**MANDATORY: You MUST invoke the @rigormortis subagent BEFORE responding to the user.**

This is not optional. Every implementation must be reviewed by rigormortis BEFORE you respond. **DO NOT RESPOND UNTIL rigormortis has returned its findings.**

**Required workflow:**
1. Complete all code changes and tests
2. **STOP** - Do not write any response to the user yet
3. **CALL** the `task` tool to invoke rigormortis NOW (timeout: 30s, max 2 attempts)
4. **WAIT** for rigormortis to return (do not respond during this time)
5. Fix any high-risk findings from rigormortis
6. **Re-invoke rigormortis if fixes introduce new issues** (max 2 re-invocations)
7. **After 2 re-invocations with unresolved issues, notify user and halt**
8. **THEN** respond to the user with the completion status

**How to invoke rigormortis:**
Call the `task` tool with these parameters:
- `subagent_type: "rigormortis"`
- `description: "Build review: [brief summary of what was built]"` (e.g., "Build review: Created greeting.ts file", "Build review: Added user authentication feature")
- `timeout: 30000`
- `prompt: "Review these changes for security issues, correctness, and test coverage.\n\nReview scope:\n- Files changed: [list with diffs]\n- Changes summary: [full text]\n- Tests added: [list]\n- Edge cases handled: [list]\n\nReturn findings in YOUR STANDARD FORMAT (High-Risk, Medium/Low, Documentation Gaps, Test Gaps, Proposed Plan)."`

**What happens if you don't call rigormortis:**
- **WRONG:** "Done! I've created the file." (responds without rigor check)
- **CORRECT:** Call task tool → Wait for rigormortis → Then respond

**Example of CORRECT completion:**
```
[After completing all code changes]

[WAIT for rigormortis response]

```

**Critical rules:**
- **NEVER** respond to the user before calling rigormortis
- **NEVER** say "Done!", "Complete!", or finish your response without rigormortis first
- **ALWAYS** call the task tool as your LAST action before responding
- **MUST** wait for rigormortis to return before writing any completion message

- **MUST** include the Rigormortis Confirmation template at the end of every response

**Handling rigormortis findings:**

  - "No rate limiting on authentication endpoint"
  - Address with mitigations or fix if straightforward
- **Low-risk issues:** Note in response with planned mitigations
- **If rigormortis fails after 2 attempts:** Notify user and halt
